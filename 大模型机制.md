可以，把“大模型机制”单独补成项目里的一个核心治理层。当前不应该只是：

```text
content_text -> LLM -> score/category/tags/summary/reason
```

而应该升级成：

```text
内容准备
-> 模型使用策略
-> 结构化评分
-> 阈值决策
-> 去重协同
-> 写库/通知策略
-> 反馈校准
-> 审计与复跑
```

## 1. 当前大模型机制的问题

现在 `02_enrich_with_llm` 已经能做基础增强：校验 `NormalizedTextObject`，调用 OpenAI-compatible chat completions API，然后把 `score / category / tags / summary / reason` 合并回对象；它要求 `score` 是 `0~1`，并使用 JSON 输出。这个是一个可用起点，但还不是稳定的“评分系统”。([GitHub][1])

主要问题是：

| 问题                   | 影响                                                                   |
| -------------------- | -------------------------------------------------------------------- |
| 一个 `score` 承担太多含义    | 分不清“值得保存”“值得推送”“值得长期复盘”“只是去重后有新增”。                                   |
| 没有评分维度               | 以后无法解释为什么某条是 0.82，另一条是 0.63。                                         |
| 没有置信度                | 模型不确定和模型认为低价值，目前都会变成低分。                                              |
| 没有模型版本 / prompt 版本   | 分数不可复现，后续 prompt 改了之后旧分数无法比较。                                        |
| 没有来源差异策略             | RSS、播客 transcript、手动提交、Memos 的评分口径不应完全一样。                            |
| 没有隐私分流               | 个人 memo、ManicTime、私域聊天内容不应该默认送外部模型。                                  |
| LLM 在去重前调用，成本偏高      | 重复内容也会先进 LLM，浪费 token。                                               |
| 长 transcript 直接进模型不稳 | 你本地已经观察到超长 transcript 在 embedding/provider 通道下不稳定，8000 字符比 12000 更稳。 |
| 没有反馈闭环               | 你后面是否真的阅读、收藏、删除，没有回流到评分机制。                                           |

---

## 2. 先明确：LLM 的职责边界

大模型不应该决定一切。建议把职责拆开。

| 环节                               | 是否交给 LLM | 原因                        |
| -------------------------------- | -------: | ------------------------- |
| `item_id`、`content_hash`、URL 规范化 |        否 | 必须确定性、幂等。                 |
| 基础字段清洗                           |        否 | 规则处理更稳定。                  |
| 内容摘要                             |        是 | LLM 适合。                   |
| 标签建议                             |    是，但受控 | LLM 可建议，但标签体系要有限制。        |
| 内容价值评分                           |  是，但要结构化 | 不能只给一个黑盒分数。               |
| 去重判定                             |      主要否 | Qdrant 相似度是主判定，LLM 只解释差异。 |
| 是否写入 Vault                       |     部分参考 | 最终应由规则矩阵决定。               |
| 是否通知                             |     部分参考 | 要结合分数、来源、去重、用户优先级。        |
| 私密内容处理                           |       谨慎 | 默认不能无脑送外部模型。              |

核心原则：

```text
LLM = 评价与解释层
Qdrant = 相似度判定层
规则引擎 = 最终动作决策层
Obsidian = 主事实库
```

## 3. 推荐把单一 `score` 拆成 4 类分数

当前 `NormalizedTextObject` 已经把 `score / category / tags / dedupe_action / status / vault_path` 放在统一对象契约里，说明评分本来就是主干 contract 的一部分。建议保留顶层 `score` 做兼容，但新增一个结构化评分对象。([GitHub][2])

### 推荐字段

```yaml
score: 82
score_scale: 100

ai_score:
  keep_score: 82
  notify_score: 64
  reference_score: 88
  action_score: 45
  confidence: 0.76
  decision_hint: keep_reference
  reason: "内容对长期 AI 信息处理系统设计有参考价值，但没有强即时行动项。"

score_dimensions:
  personal_relevance: 22
  information_density: 18
  novelty: 16
  actionability: 8
  long_term_value: 17
  source_quality: 4
  penalty: -3

ai_meta:
  model_profile: scoring_default
  provider: deepseek
  model: deepseek-chat
  prompt_version: scoring_v1
  rubric_version: personal_info_processor_v1
  input_chars: 7821
  scored_at: "2026-04-19T..."
```

### 四类分数的含义

| 字段                | 含义              | 用途                     |
| ----------------- | --------------- | ---------------------- |
| `keep_score`      | 是否值得进入 Obsidian | 决定写库。                  |
| `notify_score`    | 是否值得即时提醒        | 决定飞书/企业微信/桌面通知。        |
| `reference_score` | 是否有长期参考价值       | 决定是否放入长期知识库、是否进入复盘。    |
| `action_score`    | 是否能转成行动         | 决定是否生成 todo、项目任务、待读清单。 |
| `confidence`      | 模型对自己判断的把握      | 决定是否需要人工 review。       |

顶层 `score` 可以等于：

```text
score = keep_score
```

这样不会破坏现有 `NormalizedTextObject`。

## 4. 推荐评分维度

不要让模型自由发挥“我觉得很有价值”。要给它固定 rubrics。

建议总分 100 分：

| 维度                         |      分值 | 判断问题                                              |
| -------------------------- | ------: | ------------------------------------------------- |
| 个人相关性 `personal_relevance` |    0–25 | 是否贴合你的长期关注：AI 信息处理、自动化、知识库、n8n、Obsidian、转录、个人数据流。 |
| 信息密度 `information_density` |    0–20 | 是否有具体方法、数据、架构、代码、经验，而不是泛泛观点。                      |
| 新颖性 `novelty`              |    0–20 | 是否提供新角度，还是和已有内容高度重复。                              |
| 可行动性 `actionability`       |    0–15 | 是否能转成配置、代码、任务、流程改造。                               |
| 长期参考价值 `long_term_value`   |    0–15 | 半年后是否仍值得检索。                                       |
| 来源质量 `source_quality`      |     0–5 | 来源是否清楚、可回溯、可信。                                    |
| 惩罚项 `penalty`              | 0 到 -20 | 广告、标题党、空泛、重复、低质量转录、缺上下文、过短内容。                     |

然后算：

```text
keep_score =
  personal_relevance
+ information_density
+ novelty
+ actionability
+ long_term_value
+ source_quality
+ penalty
```

注意：`novelty` 不应只靠 LLM 感觉。最好把 Qdrant 最近邻结果传给 LLM，让它基于相似内容做“差异判断”。

## 5. 大模型调用顺序建议调整

现在主线大致是：

```text
00 normalize
-> 02 LLM enrich
-> 03 Qdrant gate
-> writer
```

这个顺序能跑，但成本和逻辑都不是最优。因为近似重复内容也会先消耗 LLM。

我建议改成：

```text
00_common_normalize_text_object
-> 01a_rule_prefilter
-> 03a_qdrant_search_and_decide
-> 02_common_llm_enrich_score
-> 05_common_vault_writer
-> 03b_qdrant_commit
```

也就是：

```text
先便宜判断
再向量去重
再决定是否值得调用 LLM
最后写库和索引
```

### 新顺序的好处

| 改动                           | 好处                          |
| ---------------------------- | --------------------------- |
| LLM 放到 Qdrant 初判之后           | 重复内容可以不进 LLM，省 token。       |
| Qdrant 只做相似度                 | 不让 LLM 替代向量去重。              |
| LLM 可以看到最近相似内容               | 能更好判断“新增价值”。                |
| writer 在 Qdrant commit 前     | 避免“Qdrant 已索引但 Vault 没写成功”。 |
| `score` 与 `dedupe_action` 分离 | 不会把“低价值”和“重复”混在一起。          |

你现在的 `.env.example` 已经把 LLM 和 embedding 分开：主链 LLM 默认是 `deepseek-chat`，embedding 默认是 `text-embedding-3-small`，Qdrant 阈值是 `0.85 / 0.97`。这说明模型使用本来就应该分层治理，而不是一个模型包办所有事情。([GitHub][3])

## 6. LLM 使用策略：不是所有内容都同等处理

建议新增一个 `llm_policy`，由规则节点生成。

```yaml
llm_policy:
  should_call_llm: true
  call_reason: "new_or_diff_content"
  model_profile: "scoring_default"
  input_strategy: "direct_8000_chars"
  privacy_level: "public"
  max_input_chars: 8000
  allow_external_llm: true
```

### 按来源区分

| 来源                       | 默认策略                      |
| ------------------------ | ------------------------- |
| RSS / YouTube XML / 公开播客 | 可以使用外部 LLM。               |
| transcript               | 可以用外部 LLM，但要截断或分块。        |
| manual URL               | 默认至少进入 review，不要因为低分直接丢弃。 |
| Memos                    | 默认视为个人内容，外部 LLM 要谨慎。      |
| ManicTime / 私人聊天 / 语音复盘  | 默认不送外部 LLM，除非显式允许。        |
| 微信群总结                    | 要区分公开群、私域群、敏感群。           |

### 按去重结果区分

| `dedupe_action` | LLM 策略                         |
| --------------- | ------------------------------ |
| `full_push`     | 正常打分、摘要、标签。                    |
| `diff_push`     | 让 LLM 重点判断“与已有内容相比新增了什么”。      |
| `silent`        | 默认不调用 LLM，只记录运行日志；可以做 5% 抽样审计。 |

当前架构文档里已经定义了三级降噪逻辑：`<0.85` 完整推送，`0.85~0.97` 增量摘要，`>0.97` 静默处理。这个机制应该继续保留，但不要让 LLM 替代这个判定。([GitHub][2])

## 7. 动作决策矩阵

分数本身不直接等于动作。建议用矩阵决定。

### 基础阈值

```yaml
thresholds:
  keep_full: 70
  keep_review: 50
  notify: 85
  action_item: 75
  low_confidence: 0.55
```

### 决策矩阵

| 条件                                              | 动作                       |
| ----------------------------------------------- | ------------------------ |
| `dedupe_action=full_push` 且 `keep_score>=70`    | 写入 Vault。                |
| `dedupe_action=full_push` 且 `notify_score>=85`  | 写入 Vault + 通知。           |
| `dedupe_action=full_push` 且 `50<=keep_score<70` | 写入 `Inbox/Review` 或轻量笔记。 |
| `dedupe_action=full_push` 且 `keep_score<50`     | 不写正文，只写运行日志。             |
| `dedupe_action=diff_push` 且 `keep_score>=60`    | 更新已有 note，追加“新增观点”。      |
| `dedupe_action=diff_push` 且 `keep_score<60`     | 不通知，只记录差异摘要。             |
| `dedupe_action=silent`                          | 默认不写新 note，不通知。          |
| `confidence<0.55`                               | 不自动丢弃，进入 review。         |
| `source_type=manual`                            | 不自动丢弃，最低进入 review。       |
| `privacy_level=private` 且无本地模型                  | 不调用外部 LLM，写入待处理队列。       |

关键点：
**低置信度不是低价值。低置信度应该进 review，而不是 silent。**

## 8. 长内容和 transcript 的处理机制

不要把长 transcript 直接塞进一个 LLM 请求。你已经遇到过长文本稳定性问题，本地事实里也明确提到 8000 字符比 12000 更稳。

建议分三档：

|                 内容长度 | 策略                     |
| -------------------: | ---------------------- |
|      `< 8,000 chars` | 直接送 LLM。               |
| `8,000–40,000 chars` | 分块摘要，再全局评分。            |
|     `> 40,000 chars` | 先抽取章节 / 时间段 / 关键词，再评分。 |

### transcript 推荐流程

```text
transcript_text
-> 清洗口癖、重复句、ASR 噪声
-> 按时间或语义分块
-> chunk summaries
-> global summary
-> scoring
-> tags/category
```

LLM 输入不要只放 transcript 正文，还要放：

```yaml
source_type: transcript
title:
speaker:
duration:
platform:
upstream_summary:
nearest_existing_summary:
qdrant_similarity:
manual_note:
```

这样模型才能判断“值得不值得保存”，而不是只做文本摘要。

## 9. 标签机制也要受控

现在 `02_enrich_with_llm` 会让模型返回 tags，并与已有 tags 合并。这个方向可以保留，但需要避免标签爆炸。([GitHub][1])

建议把标签分三类：

```yaml
tags:
  system:
    - inbox
    - signal-to-obsidian
  source:
    - podcast
    - transcript
  ai:
    - llmops
    - knowledge-base
    - n8n
```

LLM 只能生成 `ai` 标签，而且必须从允许列表或近似规范化表里选。

建议建一个文件：

```text
deploy/config/tag_taxonomy.yaml
```

示例：

```yaml
allowed_tags:
  - ai
  - llmops
  - automation
  - knowledge-base
  - obsidian
  - n8n
  - qdrant
  - rss
  - transcript
  - podcast
  - privacy
  - local-first
  - architecture
  - productivity

synonyms:
  vector-db: qdrant
  rag: retrieval
  note-taking: knowledge-base
  personal-knowledge-management: knowledge-base
```

模型返回未知 tag 时：

```text
未知 tag
-> 规范化到 allowed tag
-> 无法规范化则放入 ai_suggested_tags
-> 不直接进入正式 tags
```

## 13. 模型调用参数建议

当前 `02` 里 temperature 是 `0.2`。评分任务建议更低一点。

```yaml
model_profile:
  scoring_default:
    model: deepseek-chat
    temperature: 0.0
    max_output_tokens: 1200
    timeout_seconds: 60
    retry: 1

  summary_long:
    model: deepseek-chat
    temperature: 0.1
    max_output_tokens: 2000
    timeout_seconds: 120
    retry: 1

  private_local:
    model: local-model
    temperature: 0.0
    external_network: false
```

原则：

| 场景        |       temperature |
| --------- | ----------------: |
| 打分        |       `0` 或 `0.1` |
| 分类        |       `0` 或 `0.1` |
| 标签        |       `0` 或 `0.1` |
| 摘要        |     `0.1` 或 `0.2` |
| 改写 / 复盘文案 | `0.3+`，但不建议用于主干评分 |

## 14. 缓存机制必须加

LLM 结果要缓存，否则重复跑 workflow 会反复花 token，而且分数可能漂移。

缓存 key：

```text
llm_cache_key = sha256(
  content_hash
  + model_profile
  + model
  + prompt_version
  + rubric_version
  + input_strategy
)
```

缓存位置可以先简单放：

```text
/vault/.aip/llm_cache/
  <cache_key>.json
```

或者：

```text
/vault/.aip/llm_runs.jsonl
```

每次记录：

```json
{
  "run_id": "2026-04-19T...",
  "item_id": "...",
  "content_hash": "...",
  "cache_key": "...",
  "model": "deepseek-chat",
  "prompt_version": "scoring_v1",
  "rubric_version": "personal_info_processor_v1",
  "input_chars": 7821,
  "score": 82,
  "confidence": 0.76,
  "status": "ok",
  "duration_ms": 2841
}
```

## 15. 隐私与外部模型使用机制

这个项目叫“个人信息获取/处理”，如果后续接入 Memos、ManicTime、私域聊天、语音复盘，就必须先做隐私分层。

建议每条对象增加：

```yaml
privacy:
  level: public | semi_private | private | sensitive
  external_llm_allowed: true
  redaction_applied: false
  redaction_policy: none
```

### 推荐策略

| `privacy.level` | 外部 LLM | 默认动作               |
| --------------- | -----: | ------------------ |
| `public`        |     允许 | RSS、公开视频、公开播客。     |
| `semi_private`  |   谨慎允许 | 手动提交的公开链接。         |
| `private`       |  默认不允许 | Memos、个人语音、个人时间记录。 |
| `sensitive`     |     禁止 | 密钥、账号、健康、财务、私聊原文。  |

如果没有本地模型，私密内容可以：

```text
写入 Review
-> 不做 LLM enrich
-> frontmatter 标记 llm_status: skipped_privacy
```

不要为了自动化把所有私人内容都送到外部模型。

## 16. 反馈校准机制

评分系统一开始一定不准。不要追求 prompt 一步到位，要做反馈闭环。

### 在 Obsidian 里增加人工反馈字段

```yaml
user_feedback:
  read: true
  useful: true
  starred: false
  deleted_later: false
  corrected_score: 90
  feedback_note: "这篇对项目架构有直接帮助。"
```

### 每周跑一次校准

统计：

| 指标                        | 用途                  |
| ------------------------- | ------------------- |
| 高分但你从不看                   | 阈值可能太低，或 prompt 偏宽。 |
| 低分但你手动收藏                  | 漏判，需要调整个人相关性。       |
| `notify_score` 高但没行动      | 通知阈值太低。             |
| `reference_score` 高但后续没引用 | 长期价值维度过宽。           |
| transcript 普遍高分           | 模型可能被长文本“信息量假象”误导。  |

输出一个校准报告：

```text
/vault/System/Scoring Calibration/2026-04-26.md
```

## 18. 最小可落地版本

先别做复杂模型路由，第一版只做 6 个改动。

### 第一步：把 `score` 改成 0–100

现在 `score` 是 `0~1`。建议改成 0–100，更适合人工理解和 Obsidian frontmatter。

兼容方式：

```text
旧 score <= 1 时，自动乘以 100
新 score 直接使用 0–100
```

---

### 第二步：新增 `confidence`

模型必须输出：

```yaml
confidence: 0.0-1.0
```

规则：

```text
confidence < 0.55 -> review
```

---

### 第三步：新增 `score_dimensions`

至少先加这几个：

```yaml
personal_relevance
information_density
novelty
actionability
long_term_value
penalty
```

---

### 第四步：新增 `prompt_version`

每次写入 frontmatter：

```yaml
llm_model: deepseek-chat
llm_prompt_version: scoring_v1
llm_rubric_version: personal_info_processor_v1
llm_scored_at: ...
```

---

### 第五步：新增动作矩阵

先用简单规则：

```text
keep_score >= 70 -> write
50 <= keep_score < 70 -> review
keep_score < 50 -> skip
confidence < 0.55 -> review
manual source -> at least review
```

### 第六步：新增 `llm_runs.jsonl`

每次 LLM 调用都记录，不然后续无法调参。

## 19. 我建议直接写入架构文档的补充段

可以把下面这段作为“AI 打分与模型使用机制”放进架构文档。

```text
AI enrich 层不直接决定最终动作，只输出结构化评分、摘要、标签、理由和置信度。最终是否写入 Obsidian、是否通知、是否静默，必须由 score、confidence、dedupe_action、source_type、privacy_level 和用户策略共同决定。

LLM 输出至少包括：
- keep_score
- notify_score
- reference_score
- action_score
- confidence
- category
- tags
- summary
- reason
- prompt_version
- model_profile

score 代表内容保存价值，不代表去重结果；dedupe_action 由 Qdrant 相似度主导，LLM 只辅助解释 full_push / diff_push 内容的新增价值。

所有 LLM 调用必须记录 model、prompt_version、rubric_version、input_chars、content_hash、cache_key、score、confidence 和调用状态。私密内容默认不发送到外部模型，除非 source_type 和 privacy_level 明确允许。
```

## 最终建议

这个项目的大模型机制应该从：

```text
模型帮我打个分
```

升级为：

```text
模型基于固定 rubric 输出结构化评价；
系统基于 score + confidence + dedupe_action + privacy + source policy 做最终动作；
所有模型调用可复现、可审计、可校准。
```

短期最值得改的是 `02_enrich_with_llm`：

```text
score: 0~1
category
tags
summary
reason
```

升级成：

```text
score: 0~100
ai_score:
  keep_score
  notify_score
  reference_score
  action_score
  confidence
score_dimensions:
  personal_relevance
  information_density
  novelty
  actionability
  long_term_value
  penalty
ai_meta:
  model_profile
  model
  prompt_version
  rubric_version
  input_strategy
  scored_at
```

这样后续 RSS、transcript、manual、Memos 都能复用同一套大模型评价机制，不会各自长出一套“看起来像评分、实际口径完全不同”的隐性字段体系。
